# 01. コンセプト

## 解こうとしている問題

AIエージェント（Claude Code、Cursor、Devin、GitHubクラウドエージェント等）単体でアプリ開発を行うと、ロジックは書けるがアセット品質が毎回ガチャになる。具体的には3Dモデル、アニメーション、サウンド、UI/UX、コピーライティングといった「見た目と手触り」を司る要素が、プロジェクトごとに0から再構築され、しかも安定しない。

大手スタジオが安定して高品質を出せる本当の理由は才能ではなく、社内に蓄積された **アセットライブラリ・スタイルガイド・パイプライン・QCチェックリスト** である。つまり外部化可能な領域であり、ここを共通基盤として提供すれば、AIエージェント単体でも大手レベルの仕上がりに到達できる。

## 解法の核心

自前でアセットを工房化するのではなく、**既存APIのオーケストレーション層** として実装する。Sketchfab、LottieFiles、SF Symbols、Google Fonts、freesound、ElevenLabs等が既に「DNA」を持っているため、自分たちは **司書・翻訳家・品質検査官** に徹する。

このレイヤーをエージェントが MCP（Model Context Protocol）で常時参照することで、毎回ゼロから始まらず、プロジェクト固有のスタイル規定（Style Bible）に沿った一貫性のあるアウトプットが保証される。

## ポジショニング

「アセットの Netflix」ではなく「アセットの Spotify検索 + DJ + マスタリング」。在庫を持たず、世界中のアセットAPIを束ねて、文脈に合った組み合わせを賢く選び、統一フォーマットで配膳する。

## MVPのスコープ

最初の戦場は **App Store 向け一般モバイルアプリ**（SaaS、ツール、ライフスタイル）。

理由：Apple HIG準拠・UI/UX・アイコン・アニメーション・SE の品質差が露骨に出る領域であり、AIエージェントが最も素人っぽさを露呈する場所だから。ここで勝てればゲーム・Web・他プラットフォームへ横展開できる。

## 自前で持つもの／持たないもの

### 自前で持つ（コアIP）

1. **Style Bible**：プロジェクトごとのスタイル規定JSON
2. **選定ロジック**：候補から最適を選ぶランキング・組み合わせ最適化
3. **接着剤コード**：フォーマット変換、コード生成、プロジェクトへの注入

### 外部APIに委ねる

- アセット本体（3D / アイコン / アニメ / 音 / フォント）
- ライセンス管理
- コンテンツ生成基盤

## 思想の三本柱

1. 既存APIに乗っかってオーケストレーションに徹する
2. Style Bible をコアIPにする
3. QC Gate で素人っぽさを機械的に排除する

## 成功基準

Polish Layer 自身を使って App Store にアプリを1本リリースし、レビュー★4.5を取れること。
